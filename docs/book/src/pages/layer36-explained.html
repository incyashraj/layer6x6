Layer36 portable software, explained simply
Plain English explainer

What if software travelled like a PDF?

You do not download a different PDF for Mac, Windows, Linux, Android, and iPhone. You just open the file. Layer36 is trying to make apps feel more like that: one portable app artifact, with clear permissions, running through the right host adapter for each machine.

The normal problem

Today, an app is usually married to the operating system it was built for. A Windows app expects Windows APIs. A macOS app expects macOS APIs. A mobile app expects a completely different world.

That means developers often rewrite the same idea several times, or carry a complicated build matrix forever.

Layer36 asks a different question: what if the app talked to one common layer, and that layer translated to the host underneath?

Files already work this way
PDF One file. Different devices open it with their own viewer.
Typical app Different builds. Different APIs. Different permission behavior.

Layer36 is not saying apps are as simple as PDFs. It is saying software should have a portability layer that makes the app less tied to one host.

The Layer36 idea in one picture

App sourceThe developer writes the app in Rust, TypeScript, Go, or another language.
down
WebAssembly componentThe app becomes one portable artifact. This is the file Layer36 runs.
down
Layer36 runtimeThe local engine loads the component and connects it to Layer36 APIs.
down
UAPI + UCapUAPI is the common app API. UCap checks what the app is allowed to do.
down
Host adapterThe adapter translates approved calls to macOS, Windows, Linux, mobile, or web behavior.

Example: how `layer36-clock` becomes terminal output

App code `layer36-clock` source The sample app asks for time, locale, timezone, and stdout. In simple terms: "What time should I show, and where do I print it?"
Portable file WebAssembly component The app is compiled into `layer36_clock.wasm`, a portable artifact that the Layer36 runtime can load.
Runtime `layer36 run` The CLI starts a runtime session, loads the component, applies test time/locale flags, and connects the app to UAPI.
UAPI Time, locale, stdout The app calls Layer36 APIs instead of directly calling macOS, Windows, or Linux APIs.
UCap + adapter Check then translate Layer36 checks allowed capabilities, then the host adapter provides normalized time, locale, and output behavior.
Output Terminal result `app=layer36-clock`, `timezone=UTC`, `locale=en-US`, `date=1970-01-15 06:56`.

This is a small example, but it shows the complete shape: portable app artifact, Layer36 runtime, common APIs, capability checks, host adapter, and final output.

After Phase 2: what Phase 3 should unlock

Phase 2 proves the runtime path with CLI apps. Phase 3 should prove that the same model can power a real desktop app with a native window, user interaction, and explicit host permissions.

Layer36 desktop preview
One WebAssembly app, native host window, explicit permissions.

In simple terms: Phase 3 should move Layer36 from "a safe portable CLI app can run" to "a safe portable desktop app can open a real window and respond to users."

Current progress: the macOS prototype can create an opt-in AppKit window, attach a first draw view, collect native window callbacks, and process one non-blocking event-loop tick. The runtime can now request that AppKit prototype path by name, while the normal path still stays headless for safety.

What is Layer36 able to do today?

Layer36 is pre-alpha, but the current Phase 2 proof is real. The CLI can run WebAssembly components that call Layer36 for time, locale, files, terminal output, and a first local HTTP path.

The most important proof is not just "it runs." The important proof is that host access can be declared, granted, denied, and tested.

Technical terms, translated

WebAssemblyA portable program format. In simple words: the app becomes a file that a runtime can execute on different machines.
RuntimeThe local engine that opens and runs the app artifact. For Layer36, this is the `layer36` CLI/runtime.
UAPIUniversal API. Instead of asking macOS, Windows, or Linux directly for files or network, the app asks Layer36.
UCapUniversal Capabilities. The permission layer. It checks whether the app was granted file, network, time, or other host access.
Host adapterThe translator. It turns approved Layer36 calls into native operating system calls.
ManifestA small file that says what the app is and what permissions it needs.

What happens when an app reads a file?

1

The app asks

The WebAssembly component calls the Layer36 file API.

2

Layer36 checks

UCap checks whether this run granted `fs.read` for that path.

3

The host acts

If allowed, the adapter reads the file through the native OS. If not, the app gets permission denied.

The honest status

Layer36 is not a finished universal app platform yet. It does not have production GUI, mobile, app bundles, marketplace, or hardened third-party sandboxing.

What it does have is a credible pre-alpha runtime proof: portable components, a real UAPI slice, explicit permissions, denial behavior, sample apps, and evidence tracking.

One sentence

Layer36 is building the runtime layer that lets software run more like portable files: one artifact, explicit permissions, host adapters underneath.

Layer36 is pre-alpha. This page explains the current proof and the direction, not a finished production platform.
