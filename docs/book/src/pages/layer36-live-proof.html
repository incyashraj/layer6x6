The complete `layer36-clock` path

Use this diagram before the terminal command. It shows what happens between the sample app and the final printed date.

1 app `layer36-clock` source The app asks for time, locale, timezone, and stdout.
2 build `layer36_clock.wasm` The source becomes a portable WebAssembly component.
3 runtime `layer36 run` The CLI starts the runtime session and loads the component.
4 UAPI time + locale + stdout The app talks to Layer36 APIs instead of host-specific APIs.
5 host UCap + adapter Layer36 checks permission shape, then the adapter supplies host behavior.
6 result terminal output `date=1970-01-15 06:56` appears from a real runtime path.
Plain-English takeaway: even this tiny clock app passes through the same layers future apps will use: portable artifact, runtime, UAPI, capability model, host adapter, output.
Step 1

Build three portable app artifacts

These commands compile small Layer36 sample apps into WebAssembly. In simple terms, this is where the app becomes a portable file that the Layer36 runtime can open.

Plain-English takeaway: Layer36 is not just a document. The demo starts by producing real runnable app artifacts.
cargo build -p layer36-cli
scripts/build-layer36-clock-component.sh
scripts/build-layer36-cat-component.sh
scripts/build-layer36-curl-component.sh

layer36_clock.wasm
layer36_cat.wasm
layer36_curl.wasm
Step 2

Show that permissions are visible

The manifest is the app's permission label. It says the app wants command-line arguments, output, error output, and file read access.

The key line is launch grant needed: yes for file reading. That means file access is not assumed.

Plain-English takeaway: before an app runs, Layer36 can show what it wants from the computer.
target/debug/layer36 manifest explain apps/layer36-cat/manifest.toml

Capabilities
  - io.args
    default grant        yes
  - io.stdout
    default grant        yes
  - fs.read:fixtures/**
    default grant        no
    launch grant needed  yes
Step 3

Run a harmless capability path

The clock app uses Layer36 for time, locale, timezone, and output. It proves the component is executing through Layer36's UAPI instead of being a fake screenshot.

Plain-English takeaway: a WebAssembly app can ask Layer36 for normal host services and get a stable result.
target/debug/layer36 run \
  --auto-grant \
  --manifest apps/layer36-clock/manifest.toml \
  --test-time 1234567890 \
  apps/layer36-clock/.../layer36_clock.wasm

app=layer36-clock
timezone=UTC
locale=en-US
date=1970-01-15 06:56
Step 4

Deny file access without permission

This is the most important moment. The file app asks to read a file, but this run has not granted file access. Layer36 stops it before native file access happens.

Plain-English takeaway: the runtime can say no. That is the foundation for running generated or third-party tools safely.
layer36 run \
  --manifest manifest.toml \
  layer36_cat.wasm \
  -- ./fixtures/hello.txt

permission denied: missing required capabilities
  - fs.read:fixtures/**
Step 5

Grant access and rerun the same app

Now the same app succeeds because the host approved the declared file permission. Same app path, different permission decision.

Plain-English takeaway: Layer36 makes host access explicit instead of ambient. The app does not just inherit everything the machine can do.
layer36 run \
  --manifest manifest.toml \
  --auto-grant \
  layer36_cat.wasm \
  -- ./fixtures/hello.txt

hello from Layer36
Step 6

Show that network follows the same model

The curl sample is allowed to connect only because this run grants the local endpoint. File and network are different host powers, but the Layer36 model is the same: declare, grant, check, execute.

Plain-English takeaway: UCap is not a one-off file demo. It is the permission pattern for host capabilities.
layer36 run \
  --grant net.connect:127.0.0.1:8765 \
  layer36_curl.wasm \
  -- http://127.0.0.1:8765/demo.txt

portable runtime response

What to say while showing the terminal

Short voiceover

Layer36 is trying to make software travel more like files. The app becomes a portable WebAssembly artifact. Instead of talking directly to macOS, Windows, or Linux, it talks to Layer36. Layer36 checks what the app is allowed to do, then the host adapter performs approved work on the local machine.

Final sentence

Layer36 already runs portable WebAssembly apps through explicit host permissions; the next milestone is proving this with outside developers, cross-host evidence, and the Phase 3 desktop UI layer.
